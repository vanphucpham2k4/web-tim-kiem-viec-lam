1. Về mặt kỹ thuật:
   - TinTuyenDung.MaTinTuyenDung: int (Primary Key)
   - TinUngTuyen.MaTinTuyenDung: string (không phải FK)
   - Không đồng nhất kiểu dữ liệu (int vs string)
   - Không có ràng buộc referential integrity

2. Về mặt nghiệp vụ:
   - MaTinTuyenDung trong TinUngTuyen là cầu nối logic để biết hồ sơ ứng tuyển thuộc tin nào
   - Quan hệ này CẦN THIẾT để hiểu luồng nghiệp vụ: Ứng viên nộp đơn vào công việc nào

3. Lý do thiết kế:
   - Lưu snapshot/lịch sử: Giữ lại thông tin ứng tuyển ngay cả khi tin tuyển dụng gốc bị xóa
   - Tránh mất dữ liệu: Nếu có FK constraint, việc xóa tin tuyển dụng có thể ảnh hưởng đến lịch sử ứng tuyển
   - Tương thích dữ liệu cũ: Có thể có dữ liệu cũ với MaTinTuyenDung là string

4. Rủi ro tiềm ẩn:
   - Dữ liệu "mồ côi" (orphan): TinUngTuyen có thể trỏ đến TinTuyenDung không tồn tại
   - Khó truy vấn ngược: Không thể dùng JOIN trực tiếp, phải convert string -> int
   - Không đảm bảo tính toàn vẹn: Có thể có MaTinTuyenDung không hợp lệ

5. Khuyến nghị khi vẽ sơ đồ:
   - VẪN NÊN vẽ đường nối giữa TinUngTuyen và TinTuyenDung
   - Đánh dấu bằng nét đứt (dashed line) hoặc ghi chú "Logical Relation"
   - Ghi chú rõ: "No FK Constraint" hoặc "Soft Reference"
   - Mục đích: Thể hiện đúng logic nghiệp vụ, không chỉ cấu trúc database

2. CÁCH THỂ HIỆN CÁC LOẠI QUAN HỆ
═══════════════════════════════════════════════════════════════════════════════

A. QUAN HỆ CÓ FOREIGN KEY CONSTRAINT (Hard FK):
   - Sử dụng đường nét liền (solid line)
   - Ghi rõ cardinality: 1:1, 1:N, N:1, N:N
   - Ghi chú Delete Behavior: CASCADE, SET NULL, RESTRICT
   - Ví dụ: ApplicationUser -> UngVien (1:1, CASCADE)

B. QUAN HỆ LOGIC (Logical Relation - Không có FK Constraint):
   - Sử dụng đường nét đứt (dashed line) hoặc nét chấm (dotted line)
   - Ghi chú rõ: "Logical Relation" hoặc "No FK Constraint"
   - Vẫn ghi cardinality: 1:1, 1:N, N:1
   - Ví dụ: TinUngTuyen ──[dashed]──> TinTuyenDung (N:1, Logical)

C. QUAN HỆ KẾ THỪA (Inheritance):
   - Sử dụng mũi tên tam giác rỗng (hollow triangle arrow)
   - Ví dụ: ApplicationUser ──▷ IdentityUser

4. CÁC KÝ HIỆU KHUYẾN NGHỊ
═══════════════════════════════════════════════════════════════════════════════

PRIMARY KEY:
   - Gạch chân (underline) hoặc ký hiệu [PK]
   - Ví dụ: MaUngVien [PK]

FOREIGN KEY:
   - Ký hiệu [FK] hoặc ghi chú "FK -> EntityName"
   - Ví dụ: UserId [FK -> ApplicationUser]

UNIQUE CONSTRAINT:
   - Ký hiệu [UQ] hoặc [UNIQUE]
   - Ví dụ: UserId [UQ]

INDEX:
   - Ký hiệu [IDX] hoặc ghi chú trong phần riêng
   - Ví dụ: idx_ungvien_userid [IDX]

NULLABLE:
   - Dấu ? sau kiểu dữ liệu
   - Ví dụ: string? (nullable), string (required)

CARDINALITY:
   - 1:1 (One-to-One)
   - 1:N (One-to-Many)
   - N:1 (Many-to-One)
   - N:N (Many-to-Many) - không có trong hệ thống này

DELETE BEHAVIOR:
   - CASCADE: Xóa dữ liệu con khi xóa dữ liệu cha
   - SET NULL: Đặt FK = NULL khi xóa dữ liệu cha
   - RESTRICT: Không cho phép xóa nếu có dữ liệu con

7. LƯU Ý KHI VẼ CLASS DIAGRAM (UML)
═══════════════════════════════════════════════════════════════════════════════

- Mỗi class = 1 entity
- Attributes = Properties/Columns
- Methods = Có thể bỏ qua (chỉ tập trung vào cấu trúc dữ liệu)
- Associations = Relationships
- Navigation Properties = Thể hiện bằng mũi tên hai chiều hoặc một chiều
- Inheritance: ApplicationUser ──▷ IdentityUser

═══════════════════════════════════════════════════════════════════════════════
8. LƯU Ý KHI VẼ ERD (ENTITY RELATIONSHIP DIAGRAM)
═══════════════════════════════════════════════════════════════════════════════

- Entity = Hình chữ nhật
- Attributes = Hình oval hoặc liệt kê trong entity
- Relationships = Hình thoi hoặc đường nối
- Primary Key = Gạch chân hoặc [PK]
- Foreign Key = Ghi chú [FK] hoặc đường nối có mũi tên
- Cardinality = 1, N, hoặc số cụ thể
- Optionality = (0..1) hoặc (1..N)

═══════════════════════════════════════════════════════════════════════════════
9. LƯU Ý KHI VẼ DATABASE SCHEMA DIAGRAM
═══════════════════════════════════════════════════════════════════════════════

- Table = Hình chữ nhật
- Columns = Liệt kê trong table
- Data Types = Ghi rõ (int, string, decimal(18,2), DateTime, etc.)
- Primary Key = Gạch chân hoặc [PK]
- Foreign Key = Đường nối có mũi tên + ghi chú [FK]
- Indexes = Có thể ghi chú riêng hoặc ký hiệu [IDX]
- Constraints = Ghi chú UNIQUE, CHECK, etc.
- Logical Relations = Nét đứt + ghi chú "Logical" hoặc "No FK"
